How it works

The same logic that runs in the Python MCP servers (see mcp_servers/core.py in the repo) has been ported to vanilla JS so this page works without a backend. Every action you take in the shopper view writes to the same in-memory ledger that the brand view reads.

Auction

rank_score = bid_cpc × relevance. Top-K wins by rank score. The winner's cleared CPC is (runner_up.rank_score / winner.relevance) + $0.01, capped at the winner's bid — the same generalized second-price (GSP) mechanic used by Google Search Ads and Amazon Sponsored Products.

Disclosure

Every sponsored payload carries sponsored: true and a human-readable "Sponsored — paid placement" string. The shopper-side agent is contractually required to surface it verbatim. You'll see it on every sponsored card above — it's emitted by the protocol, not the UI.

Privacy

No PII passes through the auction. Targeting is keyword and category only (contextual). Brand-side data is scoped to the authenticated brand. The protocol intentionally has no hooks for behavioural targeting in v1.

Try this

  1. Search "brightening vitamin c" — Luma Glow's LG-003 should win slot 1 over Aurora Botanica.
  2. Add the top sponsored product to your cart, then check out. ROAS appears under the brand console.
  3. Go to the brand console, raise Luma Glow's bid to $2.40, and run the forecast.
  4. Search the same query again — note how the cleared CPC changes when the runner-up gets closer.

Want the real, MCP-protocol server? It's in mcp_servers/shopper_server.py and mcp_servers/brand_server.py. Install with pip install "mcp[cli]", then run python -m mcp_servers.shopper_server.
